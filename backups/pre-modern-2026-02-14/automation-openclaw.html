My experience with OpenClaw

After two weeks using (and honestly wrestling with) OpenClaw, I can finally say I have a solid, working setup. In this post I’ll share the key lessons from the process, the main configurations (including the models I’m running), what I’m using it for today, and the next steps I’m taking.

  • Sandboxed Setup
  • Multi-model Routing
  • Telegram Connected
  • Task Automation

So… what is OpenClaw?

Before it was called Moltbot, and it went through a few names, but in simple terms it is an agent running at computer level. The easiest way to explain it is this:

It is an AI assistant that does not just answer you. It can actually act inside your machine.

A slightly more technical but still simple explanation would be this. OpenClaw is a runtime that connects a language model with tools and system access. The model thinks. The runtime interprets. The tools execute. And the workspace defines how everything behaves.

How it is built (in layers)

It is built in layers. You have the model, the runtime, the gateway, the workspace, and the tools. I know it can sound like a lot, but if you see it as a stack, it becomes very intuitive.

Here is the simple version of what each layer does:

  • The model is the brain. You can connect different models, either via API (OpenAI, Anthropic, etc.) or locally using something like Ollama. Later I’ll explain the tradeoffs and what I’m using right now.
  • The runtime is the engine. It sends the context and conversation to the model, reads the structured output, decides when to call tools, and applies the rules you define.
  • The gateway keeps the agent alive and reachable. It lets you connect channels like Telegram and talk to your assistant while it runs in the background.
  • The workspace is where everything lives: identity files, rules, tools, context documents and logs. These documents are critical because every message you send also includes these docs as input tokens to the model.
  • The tools are what make this real. Without tools, it is just chat. With tools, it can access files, send emails, create calendar events, modify sheets, browse the web, and more.

Model

Reasoning and planning engine that generates structured actions.

Runtime

Executes policy, dispatches tool calls and orchestrates every turn.

Gateway

Keeps the assistant reachable from channels such as Telegram.

Workspace

Persistent context: identity, rules, memory docs and logs.

Tools

Execution layer for files, web, productivity apps and automations.

If I had to use a simple analogy: the model is the brain, the runtime is the nervous system, the workspace is memory and personality, the tools are the hands, and the gateway is the phone line.

How everything connects

This is the mental model that helped me the most:

YOU (Telegram / CLI)
↓
GATEWAY
↓
RUNTIME (rules + memory)
↓
MODEL (Haiku / Opus / Local)
↓
TOOLS (Gmail, FS, etc.)
WORKSPACE (Context docs)

Installation mindset

There are basically three ways to run this:

  • On a remote server
  • On your own machine
  • On your machine but dockerized

In my case, I installed it directly on my machine. Bare metal. Yes, maybe slightly crazy. But it was intentional. My mindset was lean and UAT focused. I did not want to spend 1000 hours building a perfect infrastructure for something I did not fully understand yet. I wanted to test fast and learn fast. If it did not fit my workflow, I could stop without having overinvested in setup.

That said, I wasn’t stupid either. And here comes the important part.

Wait, what is the danger?

When you run an agent locally, you are giving it access to your system. Your files. Your data. Whatever permissions you decide to grant. A bad execution could cost you.

So I went very slowly.

Sandbox first

At the beginning I gave access to one single folder and started with read only. Then limited write access. Then very small, very controlled tasks. I invested a lot of time writing detailed context documents with clear boundaries and rules. This is extremely important because those documents are sent as tokens to the model on every interaction. Bad instructions mean unpredictable behaviour.

I also did stress testing. Real ones. I asked it to do things it was not allowed to do. I tried to push it outside the sandbox. I even tried begging it to ignore constraints just to see how it reacted. Honestly, in general, I was pleasantly surprised. Not perfect, but much better than I expected.

On top of that, I added contingency measures, just in case:

  • Backups of important folders
  • Version snapshots of the workspace
  • Logs for key actions
  • Incremental permission expansion, never full access from day one
Step 1Read-only access to one controlled folder.
Step 2Limited write permissions for low-risk tasks.
Step 3Stress tests against defined boundaries and rules.
Step 4Contingency setup with backups, snapshots and logs.
Step 5Incremental expansion only after reliable behavior.

Model selection: API vs local

This part took time. I tested several local models and several API models. The one I liked the most was Anthropic, especially Opus. It is honestly impressive. But there is one thing you have to keep in mind: price. These models are expensive, and using them via API is significantly more expensive than having a standard subscription.

After many variations, I decided to go for a multi model setup. My current stack looks like this:

  • Base model: Haiku. It is not huge in terms of parameters, but it is very agile, works well with tools and supports vision. For me, vision was important. I wanted the agent to understand images and extract useful info from them. It is also much cheaper, roughly three times cheaper than Sonnet. For daily tasks, it is perfect.
  • Critical layer: Opus. This one is reserved for complex reasoning, long context tasks and code heavy work. It is powerful, but I only use it when really necessary.
  • Local fallback: Ollama. Setting this up was not easy. Running a local model sounds great, but your hardware carries everything. The advantages are clear: no internet needed, no API cost, offline backup. The downsides are also clear: you need a strong machine, it consumes resources and disk space. One key lesson here is to check that the model has the tools tag in the Ollama library. Without tool support, it simply will not work properly as an agent.
Model Best for Cost profile Tradeoff
Haiku Daily tasks, tool calling, vision-supported flows Low Less depth for very complex reasoning
Opus Long-context analysis and code-heavy problem solving High Use selectively to control spending
Ollama (local) Offline fallback and privacy-sensitive scenarios No API cost Hardware load and setup complexity

What do I use it for?

As I said before, everything started in a sandbox. Only after passing tests did I expand. Here are some of the main things I do today.

Telegram Operations

Remote control of tasks while the gateway keeps the assistant active in background.

File Intelligence

Project analysis, folder organization and document-level insights.

Productivity Tracking

Habit trends and time-based analysis with actionable summaries.

Recurring Tasks

Scheduled checks, logs and periodic review loops with automation logic.

Google Integrations

Email, calendar, Sheets and Drive actions under gradual access control.

Web Structuring

Navigation, scraping and structured data extraction for downstream workflows.

Communication via Telegram

The process is simple. You create a bot, validate the token, connect it to OpenClaw. As long as the gateway is running, I can talk to my assistant from anywhere. There is something very powerful about being at the gym, sending a task to your assistant, and knowing it is executing while you train.

Analysis and organization of my computer

I gave access very gradually. Now it understands my project structure, my automation experiments and even my interests. I can ask it to reorganize folders, analyze documents, extract insights or suggest productivity improvements based on what it sees.

Productivity analysis

For the last few weeks I have been working on a structured productivity system where I track time spent on work and personal tasks, goals and counters. Axis, which is the name of my assistant, analyzes trends, compares time ranges and extracts insights about habits and evolution. Some of the insights have been genuinely useful.

Recurring tasks

One of my favorite features is scheduling recurring intelligent tasks. For example:

  • Once a week it checks for new versions of OpenClaw, compares them with mine and analyzes potential risks.
  • Daily it generates structured logs of key learnings and actions from our sessions.
  • Every fifteen days it reviews asset prices in a portfolio and extracts insights about growth or decline.

The possibilities here are almost infinite.

Gmail, Calendar and Drive

Using the Google tools integration, it can analyze emails, draft responses, send them, create events, modify Sheets and create Drive files. But again, this is sensitive. So I applied the same sandbox logic: activate one app at a time, read first, write later, test copies, stress test, and only then expand.

Web navigation

It can browse, scrape and structure data. Imagine the potential when that is combined with automation and scheduled tasks.

Security

Not everything is perfect. There have been important security concerns, especially around prompt injection and third party extensions. Weaker models are more vulnerable. Also, some community skills or tools may not be secure.

My personal opinion is simple. Use official tools from the OpenClaw author whenever possible. Be extremely cautious with external extensions.
